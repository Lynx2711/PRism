docker run -d -p 6379:6379 redis //this is what we usedin terminal for jobboard
```

That works. But you typed that command from memory, and if someone else cloned your project they'd have no idea what command to run. The `docker-compose.yml` file is just a way to **write that command down** in a file so anyone can run `docker compose up` and get the exact same containers, with the exact same ports, every time.

Think of it like this:

`docker run -d -p 6379:6379 redis` = cooking from memory

`docker-compose.yml` = a written recipe anyone can follow
For CodeSense we need both Redis AND Postgres running. Without compose, you'd need to remember two separate `docker run` commands. With compose, one file describes both, and `docker compose up -d` starts them both together. That's the entire value.

self in Celery tasks vs __self__
They're related but different contexts.
__self__ is Python's internal reference that exists on bound methods automatically. You never write it yourself.
self in a Celery task with bind=True is Celery explicitly passing the task instance to you as the first argument — just like how Django class-based views pass self to get() and post(). You're saying "give me a handle to this task object so I can call methods on it."
Without bind=True:
@shared_task
def analyze_pull_request(event_id):
    # no self — you're just a plain function
    # can't do self.retry(), can't check self.request.retries
With bind=True:
@shared_task(bind=True)
def analyze_pull_request(self, event_id):
    # self is the task instance
    # self.retry() — retry this task
    # self.request.retries — how many times have we tried
    # self.max_retries — the limit you set




A GitHub App has three levels of identity:
```
Level 1 — The App itself
"I am CodeSense, registered with GitHub, App ID 123456"
Proved by: your private key (.pem file)

Level 2 — The Installation
"CodeSense is installed on Lynx2711's account"
Proved by: installation ID (GitHub gives you this)

Level 3 — The actual API calls
"CodeSense, installed on Lynx2711, wants to read repo X"
Proved by: a short-lived token generated from Level 1 + Level 2
Why three levels? Because the same GitHub App can be installed by thousands of different users. When you post a review comment, GitHub needs to know not just "which app" but "which installation of that app on which account." The token you use for API calls is scoped to one specific installation — so CodeSense installed on Company A cannot accidentally access Company B's repos.
Now the code makes sense:
pythonauth = Auth.AppAuth(

)
This is Level 1 — proving you are the CodeSense app using your private key. AppAuth creates a JWT token signed with your .pem file. GitHub trusts this because only you have that private key.

The problem with stateless code review
Every tool that exists today — GitHub Copilot suggestions, basic linters, even most AI review tools — reviews each PR in complete isolation. They have no memory.
This means they can't answer questions like:

"This function is 80 lines — is that unusual for this codebase or normal?"
"This pattern was flagged 3 times last month and always fixed — flag it with higher confidence"
"These two files always change together — if one changed without the other, something might be wrong"
"This developer consistently writes functions without docstrings — personalize the feedback"

CodeSense answers all of these because it builds a graph of the repository over time. Every PR that gets reviewed adds to the graph. The longer it's used, the smarter the reviews get.
